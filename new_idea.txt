Title: Position Re-normalizing KV Cache Transfer

Main Idea: Treat KV as coordinate-bound state and allow position shifting (Δ) during transfer so cached prefixes can be reused at different absolute offsets (RoPE).
Basically, Enable reuse of cached KV for the same token sequence at a different absolute offset (by RoPE renormalization during transfer/import). 
Existing systems either (i) disaggregate prefill/decode and optimize KV transfer overhead, or (ii) reuse KV only when the prefix matches exactly; 
none treat KV as position-transformable state enabling reuse under offset shifts in RoPE models.

Benefits: 
  - Reuse the same KV prefix even when it appears later in a conversation / longer context
  - More flexible decode migration / scheduling because absolute token indices no longer “pin” KV

Implement:
  - custom KVConnector that: 
    * On save/export (prefill side): writes KV as usual, plus metadata {pos_base, rope_theta, layout}.
    * On load/import (decode side): applies a fast GPU kernel that rotates K by 𝑅(Δ) (and any needed layout transform) before placing into the paged KV buffers.

  - Needed changes: 
    % bind_connector_metadata(...): compute Δ per request/sequence.
    % register_kv_caches(...): grab pointers to paged KV tensors.
    % wait_for_layer_load(layer): correctness barrier if your Δ-rotation is async.
    % wait_for_save(): ensure exports don’t race KV reuse.

Measure: Constructs prompts where prefix appears at different offsets
  - prefill tokens recomputed
  - e2e latency
  - TTFT
  - correctness

Plan: 
  1. Add metadata plumbing 
     - Extend connector metadata to carry: pos_base_src, pos_base_dst, delta, rope_params
     - Make the decode side compute Δ from “where this prefix will sit” vs “where it was computed.”
  2. Implement Δ-rotation kernel
     - Kernel that applies RoPE rotation to K only (that’s the direct positional anchoring).
     - We can discuss whether V needs adjustment—usually not directly for RoPE, but there are caveats.
  3. Integrate into load path
     - Load KV blocks into a staging tensor (or directly into paged KV if layout matches)
     - Apply Δ-rotation
     - Mark layer as “loaded”
  4. One minimal correctness test
     - Single request, fixed seed, greedy decode: baseline vs disaggregated+renorm match.
  5. One minimal novelty benchmark
     - Prefix reused at different offsets → show prefill compute saved + TTFT reduction.


Title: Inline KV Transform During Transfer
Main idea: Hide work during KV transfer, such as quantizing, compressing, and pruning.
  - quantize KV (fp16→int8 or fp8-like)
  - compress (e.g., per-block delta/entropy coding)
  - prune heads / layers selectively (policy-driven)

Benefits: 
  - reduce network bytes or GPU memory footprints
  - Turn “KV movement” into a place to do useful work.

Implement: 
  - custom KVConnector that: 
    * Prefill side: pack/quantize/compress while exporting
    * Decode side: decompress/dequantize while importing into paged KV buffers

  - CUDA extension or Triton kernel for quant/dequant (fast)
  - policy hook: per-layer / per-head decisions

Measure: 
  - baseline transfer vs. quantized transfer
  - TTFT + decode throughput


Title: Demand-paged KV Transfer
Core idea: transfer KV layer-by-layer or window-by-window, begin decoding while deeper layers’ KV is still in flight

Benefits: 
  - Overlap compute with transfer
  - Reduce tail latency when only a subset is needed (e.g., short decode bursts)

Implement: 
  - On decode: start compute and call wait_for_layer_load(layer) inside attention only when that layer’s KV is required.
  - On prefill: export can be pipelined per layer.

Measure: 
  - TTFT reduction with overlapping early-layer decode compute with late-layer KV arrival.
  - timeline trace of the overlap

3) Turn the baselines into 3–5 testable hypotheses (Week 3)

You want hypotheses that are:

measurable on 2 GPUs

implementable inside/around vLLM

plausibly publishable

Here are the best candidates for your setup.

Hypothesis A — “KV transfer is dominated by copies + packing, not raw network”

Why plausible: GPU KV typically must be staged through host memory; naive code does extra copies and format conversions.

Investigations:

profile: GPU→CPU copy time, CPU packing time, gRPC serialize time

try: pinned buffers, fewer copies, larger batched sends, zero-copy-ish gRPC (where possible), different RPC transport (grpc vs tcp)

Success metric: reduce prefill_end → decode_start gap.

Hypothesis B — “KV precision/format is overkill for transport”

KV cache is big. If you can reduce bytes without tanking quality, you win.

Variants to test (in increasing ambition):

KV quantization for transport only (e.g., FP16→INT8/FP8-like with per-head scales)

Layer skipping / selective KV (send only subset of layers early, stream rest)

Chunked KV paging (send the parts decode needs first)

Success metric: lower TTFT and/or higher throughput at same p95.

Hypothesis C — “First-token placement matters more than people assume”

If the first token is generated on the decode node, you might avoid a critical stall.

Two designs:

Prefill produces KV and immediately ships; decode generates token 1

Prefill generates token 1 and ships KV + token 1

Measure TTFT difference and pipeline bubbles.

Success metric: lower TTFT at moderate concurrency.

Hypothesis D — “Overlapping KV transfer with compute can eliminate the disaggregation bubble”

Even with the same bytes, if you overlap:

late-stage prefill compute with early-stage KV transfer

early decode compute with tail KV arrival

you can reduce end-to-end latency.

Mechanisms:

stream KV per layer or per block (don’t wait for entire KV)

decode begins once it has “enough” KV for early layers (or vice versa)

Success metric: reduce idle time on decode GPU between requests.

Hypothesis E — “RAG changes the optimal split point”

RAG creates a two-phase prompt:

user query (short)

retrieved context (variable, sometimes large)

You can explore:

prefill query on A, fetch docs while transferring partial KV, then “extend prefill” without restarting everything

cache prompt prefixes (system prompt, templates) across requests

Success metric: TTFT stability under variable context sizes.
