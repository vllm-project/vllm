# Test Selection Instructions

You are selecting which tests to run for a PR based on the changed files.
Your goal: select every test that could be affected, but avoid running
unrelated tests.

## How to Use This File

You will receive:
1. A list of changed files in the PR
2. An auto-generated mapping table (source path → test directories)

Use BOTH the mapping table AND the rules below to decide which tests to run.

## Rules

### Rule 1: Use the mapping table as the baseline

The mapping table is generated by static import analysis. If a source file
is in the mapping, run all test directories listed for it. Match at the
directory level — if `vllm/attention/foo.py` changed and the mapping has
`vllm/attention/`, run every test directory listed for `vllm/attention/`.

### Rule 2: Tests not in the mapping (manual entries)

These tests don't import vllm directly (they use subprocess or HTTP clients).
Select them based on the source area changed:

| Test file | Run when these change |
|---|---|
| `tests/benchmarks/test_bench_startup.py` | `vllm/entrypoints/`, `vllm/engine/` |
| `tests/benchmarks/test_latency_cli.py` | `vllm/benchmarks/`, `vllm/entrypoints/` |
| `tests/benchmarks/test_throughput_cli.py` | `vllm/benchmarks/`, `vllm/entrypoints/` |
| `tests/cuda/test_platform_no_cuda_init.py` | `vllm/platforms/` |
| `tests/entrypoints/openai/responses/` | `vllm/entrypoints/openai/` |
| `tests/v1/kv_connector/nixl_integration/` | `vllm/v1/kv_connector/` |
| `tests/v1/spec_decode/test_backup_token_async_spec.py` | `vllm/v1/spec_decode/` |
| `tests/basic_correctness/` | `vllm/forward_context.py` (used in every forward pass, narrow import surface) |

### Rule 3: Non-Python file changes

| File type | Tests to run |
|---|---|
| `pyproject.toml` | `tests/basic_correctness/` |
| `requirements/*.txt` | `tests/basic_correctness/` |
| `CMakeLists.txt`, `setup.py` | `tests/basic_correctness/`, `tests/kernels/` |
| `csrc/` (C/C++/CUDA source) | `tests/kernels/` |
| `docs/` only | No tests needed |
| `*.md` only | No tests needed |
| `.buildkite/` only | No tests needed |

### Rule 4: When in doubt, include it

If a changed file doesn't clearly map to any test area, include
`tests/basic_correctness/`. It's better to run a few extra tests
than to miss a regression.

### Rule 5: Large PRs

If the PR touches more than 20 files across 5+ source directories,
run `tests/basic_correctness/` and `tests/v1/e2e/` in addition to
the mapped tests. Large PRs have higher risk of cross-cutting breakage.

## Output Format

CRITICAL: Your entire response must contain ONLY lines in this exact format:

    <test_path> | <reason>

Where:
- `test_path` is the test directory or file to run
- `reason` is a short explanation (under 15 words) of why it was selected

If no tests need to run, output exactly one line:

    NONE | <reason why no tests are needed>

Example output for a typical PR:

    tests/kernels/attention/ | mapping: vllm/v1/attention/ changed
    tests/v1/attention/ | mapping: vllm/v1/attention/ changed
    tests/basic_correctness/ | mapping: vllm/config/ changed

Example output for a docs-only PR:

    NONE | only docs/ and .md files changed (Rule 3)

Do not include any other text, explanation, markdown, or code blocks.
Only output lines in the format above.
