# Test Selection Instructions

You are selecting which tests to run for a PR based on the changed files.
Your goal: select every test that could be affected, but avoid running
unrelated tests.

## How to Use This File

You will receive:

1. A **pre-filtered candidate table** — for each changed vllm source file,
   the test files that import it (generated by static import analysis)
2. A list of **all changed files** in the PR (including non-Python files)
3. The **actual diff content** showing what changed in each file

Start from the candidate table, then use the diff to narrow down.

## Rules

### Rule 1: Candidates are the starting set, diff narrows them

The candidate table lists every test that *could* be affected based on
import analysis. It is pre-filtered to only the changed source files —
you do not need to do any lookup yourself.

Read the diff to understand **what specifically changed** and drop tests
from the candidate set that are clearly unaffected:

- **Comment-only or docstring-only changes**: no tests needed for that file
- **Change to a specific class/function**: only keep tests that exercise
  that class/function, not every test that imports the module
- **Import reordering, type hint additions, log message changes**: low risk,
  only `tests/basic_correctness/` if anything
- **Renaming an internal variable**: only tests that directly test that
  code path

If the diff is truncated or you cannot determine the scope of a change,
fall back to including all candidate tests for that file (safe default).

### Rule 2: Tests not in the mapping (manual entries)

These tests don't import vllm directly (they use subprocess or HTTP clients).
Select them based on the source area changed:

| Test file | Run when these change |
|---|---|
| `tests/benchmarks/test_bench_startup.py` | `vllm/entrypoints/`, `vllm/engine/` |
| `tests/benchmarks/test_latency_cli.py` | `vllm/benchmarks/`, `vllm/entrypoints/` |
| `tests/benchmarks/test_throughput_cli.py` | `vllm/benchmarks/`, `vllm/entrypoints/` |
| `tests/cuda/test_platform_no_cuda_init.py` | `vllm/platforms/` |
| `tests/entrypoints/openai/responses/` | `vllm/entrypoints/openai/` |
| `tests/v1/kv_connector/nixl_integration/` | `vllm/v1/kv_connector/` |
| `tests/v1/spec_decode/test_backup_token_async_spec.py` | `vllm/v1/spec_decode/` |
| `tests/basic_correctness/` | `vllm/forward_context.py` (used in every forward pass, narrow import surface) |

### Rule 3: Non-Python file changes

| File type | Tests to run |
|---|---|
| `pyproject.toml` | `tests/basic_correctness/` |
| `requirements/*.txt` | `tests/basic_correctness/` |
| `CMakeLists.txt`, `setup.py` | `tests/basic_correctness/`, `tests/kernels/` |
| `csrc/` (C/C++/CUDA source) | `tests/kernels/` |
| `docs/` only | No tests needed |
| `*.md` only | No tests needed |
| `.buildkite/` only | No tests needed |

### Rule 4: When in doubt, include it

If a changed file doesn't clearly map to any test area, include
`tests/basic_correctness/`. It's better to run a few extra tests
than to miss a regression.

### Rule 5: Large PRs

If the PR touches more than 20 files across 5+ source directories,
run `tests/basic_correctness/` and `tests/v1/e2e/` in addition to
the mapped tests. Large PRs have higher risk of cross-cutting breakage.

## Output Format

CRITICAL: Your response must have exactly two sections separated by `---`.

### Section 1: Reasoning

A short analysis (3-8 sentences) explaining:

- What the PR changes (which files, which functions/classes)
- Why you selected the tests you did (what dependency or rule applies)
- Why you excluded tests from the candidate set (if you narrowed it down)

### Section 2: Test list

After the `---` separator, output ONLY lines in this exact format:

    <test_path> | <reason>

Where:

- `test_path` is the test directory or file to run
- `reason` is a short explanation (under 15 words) for quick verification

If no tests need to run, output exactly:

    NONE | <reason why no tests are needed>

### Example output for a typical PR

    This PR modifies FlashAttention.forward() in vllm/v1/attention/flash_attn.py,
    changing how KV cache is sliced during prefill. The candidate set includes
    all tests that import vllm.v1.attention. I'm keeping the attention-specific
    tests and basic_correctness since this is a core code path. Dropping unrelated
    candidates like tests/lora/ and tests/quantization/ since the diff only touches
    attention logic.
    ---
    tests/kernels/attention/ | directly tests FlashAttention.forward()
    tests/v1/attention/ | v1 attention tests exercise this code path
    tests/basic_correctness/ | core attention code path changed

### Example output for a docs-only PR

    This PR only modifies markdown files under docs/. No Python source files
    changed, so no tests are needed.
    ---
    NONE | only docs/ and .md files changed (Rule 3)

Do not include any other text, markdown formatting, or code blocks outside
of the two sections described above.
