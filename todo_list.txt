## rwkv7.py 细化盘点（2026-07-03）

### 文件结构概览
- 顶层辅助函数：activation / token shift / cache-all 边界与 checkpoint helper / custom-op 包装
- 基础模块：
  - `RWKV7LoRA`
  - `RWKV7GroupNorm`
  - `RWKV7FeedForward`
- 核心注意力模块：`RWKV7Attention`
- 核心 block：`RWKV7Block`
- 顶层模型：`RWKV7Model`
- 顶层 CausalLM：`RWKV7ForCausalLM`

### 更细分类
#### A. 作为首个 upstream-ready core PR，大概率应保留
1. RWKV7 基础结构与参数接线
   - `RWKV7LoRA`
   - `RWKV7GroupNorm`
   - `RWKV7FeedForward`
   - `RWKV7Attention` 中非 cache-all / 非 checkpoint 的基本投影与输出逻辑
   - `RWKV7Block` 的基本 residual + norm + attn + ffn 结构
   - `RWKV7Model` / `RWKV7ForCausalLM` 的 embedding / lm_head / PP 接线
2. 最小状态缓存接口
   - `get_state_dtype`
   - `get_state_shape`
   - `get_mamba_state_dtype_from_config`
   - `get_mamba_state_shape_from_config`
   - `get_mamba_state_copy_func`
   - `SupportsMambaPrefixCaching` / `HasInnerState` / `IsAttentionFree` 这组接口声明
3. 明确的首版限制
   - `config.attn is not None` 时直接 `NotImplementedError`
   - `value_dim` 按层变化时直接 `NotImplementedError`
   这些限制有利于把首个 PR 范围收窄到“最小纯 RWKV7 core support”。

#### B. 可能保留，但需要 justification 或后续简化
1. 统一使用 `RWKV7_RUNTIME_DTYPE = torch.float32`
   - 这更像数值稳定性要求，而不是单纯优化。
   - 如果 correctness 依赖它，可以保留；但需要在 PR 说明里明确原因。
2. `v_first` 在层间和 PP 间的传递
   - 这看起来是 RWKV7 架构本身需要，不像可选优化。
   - 因此大概率应保留。
3. `static_forward_context` 注册与 block-level custom op
   - 仓库中已有类似模式（如 `olmo_hybrid.py`, `bailing_moe_linear.py`）。
   - 但 RWKV7 现在同时有 `rwkv7_attention` 和 `rwkv7_block_forward` 两层 custom op，复杂度偏高。
   - 首个 upstream PR 更适合论证“是否真的需要保留两层 custom op”，否则 reviewer 容易认为过重。

#### C. 很适合从首个 upstream PR 中后置/剥离
1. cache-all 相关辅助函数
   - `_rwkv7_cache_all_boundary_positions`
   - `_rwkv7_collect_recurrent_checkpoint_states`
   - `_rwkv7_checkpoint_offsets_from_counts`
2. `RWKV7Attention` 中 cache-all / checkpoint 发射路径
   - `forward_prefill_cache_all`
   - `forward_prefill_batch_cache_all`
3. `RWKV7Block` 中 cache-all 专门分支
   - `_run_prefill_sequence_cache_all`
   - `_run_prefill_batch_cache_all`
   - `_forward_runtime` 中 `cache_all = ...` 以及其后整段 cache-all 特化逻辑
4. 与上述逻辑配套的 metadata 扩张
   - 当前 `linear_attn.py` 为 RWKV7 增加的多组 block index / num_computed_tokens 字段，主要就是为了支撑 cache-all 路径。
   - 这类改动很像第二阶段优化，不像首个 core support 必需项。
5. RWKV7 专用 env var 开关
   - `RWKV7_DISABLE_FUSED_PREFILL`
   - `RWKV7_DISABLE_FUSED_RECURRENT`
   - 这类模型私有环境变量通常不适合作为首个 upstream 核心 PR 的一部分。
6. config 中的策略覆盖
   - prefix caching 默认强制 `align`
   - post optimization 默认把 cudagraph 从 `FULL_AND_PIECEWISE` 改成 `PIECEWISE`
   - 这些都更像 workaround / rollout policy，而不是 core model definition。

#### D. 当前最值得怀疑、应该优先再审的部分
1. `rwkv7_attention` 这层 custom op 是否必要
   - 当前 `RWKV7Block` 本身已经有 `rwkv7_block_forward` 包装。
   - 如果 block 级 custom op 足够，则 attention 级 custom op 可能是重复复杂度。
2. packed prefill 是否必须在首个 PR 就上
   - 当前 `_rwkv7_packed_prefill_enabled()` 同时影响 prefill 分支与 fused recurrent 选择。
   - reviewer 很可能更愿意先接收“逐序列 correctness 路径”，再单独上 batched/packed prefill 优化。
3. `vllm/config/compilation.py` 中追加 `vllm::rwkv7_block_forward`
   - 如果首个 PR 不保留 block-level custom op / piecewise graph 依赖，则这部分可一并后置。

### 测试文件的对应拆分建议
首个 upstream-ready core PR 更适合优先保留：

如果目标是“更像 vLLM upstream 会先接受的版本”，当前更建议首个 PR 逼近以下范围：
- HF config + registry + model registry
- 最小 RWKV7 core block/model/CausalLM 接线

  - 也就是去掉 `cache_mode == "all"` 的这套 RWKV7 metadata 膨胀；
  - 通用 backend 不应为单模型的临时优化路径背额外复杂度。

### 对首个 upstream PR 的建议
- **建议移除** `linear_attn.py` 中这组 RWKV7 专属 metadata 扩展。
- 同步要求：`rwkv7.py` 中依赖这些字段的 cache-all 分支也应一起移除 / 延后。

## config.py 细化盘点（2026-07-03）
目标文件：`/mnt/data/Codes/RWKV/vllm/vllm_rwkv7/vllm-upstream/vllm/model_executor/models/config.py`

### 现状拆解
当前新增的 `RWKV7ForCausalLMConfig` 主要做了两件事：
1. `verify_and_update_config()`
   - 当 `enable_prefix_caching=True` 且用户未显式设置 `mamba_cache_mode`（即仍为 `none`）时，先强制改成 `align`；
   - 然后再调用通用 `MambaModelConfig.verify_and_update_config()`。
2. `apply_post_optimization_level_defaults()`
   - 试图把 `FULL_AND_PIECEWISE` 强制降成 `PIECEWISE`。

### 进一步判断
- 第 1 点本质上是 **临时 runtime policy / workaround**，不是 RWKV7 架构接线本身的必要组成。
- 它存在的根因是：当前 `RWKV7ForCausalLM` 声明了 `SupportsMambaPrefixCaching`，于是通用 `MambaModelConfig` 会把 prefix caching 默认模式推到 `all`；
  但本分支里真正完整依赖 `all` 模式的，是后面那套 cache-all / checkpoint emission 路径，而这套恰恰是首个 upstream PR 最应该后置的部分。
- 第 2 点目前看起来还是 **未接线的死代码**：全仓库检索 `apply_post_optimization_level_defaults`，只有定义，没有调用方。

### 一个更干净的首版收敛方案（更推荐）
如果首个 upstream PR 目标是“先上最小 core support，再补 cache-all / 专项优化”，则更合理的收敛方式是：
1. **首版先不声明 `SupportsMambaPrefixCaching`**；
2. `MODELS_CONFIG_MAP["RWKV7ForCausalLM"]` 直接回落到通用 `MambaModelConfig`；
3. 这样当开启 prefix caching 时，通用逻辑会自然选择更保守的 `align` 路径，而不是 `all`；
4. 等后续单独补齐 cache-all / checkpoint emission 后，再考虑把 RWKV7 重新声明为 `SupportsMambaPrefixCaching`。

### 这样做的好处
- 可以直接删除 `RWKV7ForCausalLMConfig` 这整个 RWKV7 专属 policy 类。
- 可以顺带删除：
  - `config.py` 中 `CUDAGraphMode` 的 RWKV7 专属 import / override；
  - 测试里针对 `RWKV7ForCausalLMConfig` 的临时策略测试；
  - 那个当前没有调用方的 `apply_post_optimization_level_defaults()`。
- 这样更符合 upstream 习惯：
  - **模型结构接线** 和 **runtime 策略 workaround** 分开；
  - 不在首个模型接入 PR 里塞入“为了绕过当前实现缺口而加的默认策略覆盖”。
