1. 摘要

我们把 huangyibo/vllm@421fedc（"Add MooncakeStore Dummy Client for KV cache offload"）所引入的 dummy-client 支持移植到当前分支，并修复了 3 处随该 commit 一起提交的 bug。但仅做"配置接通"是不够的——直接将 GPU KV cache 指针交给 dummy client 会被拒绝。我们参考 SGLang 的设计思路，给 vLLM 的 MooncakeStoreWorker 加了一层 SHM 中转环（staging ring）：所有块通过 dummy client 自带的共享内存池中转，再加 cudaHostRegister 锁页 + cudaMemcpyAsync 单流同步的优化，把每块 stage 成本从 369 µs 降到 88 µs。

在 Llama-3.1-8B-Instruct（input 8k × 3 轮 × 50 conv）这套不会真正触发外部命中的 workload 上，dummy 模式相对 real-client 的 吞吐量回归仅 −3.8%，p99 TTFT +2.4%，整体走在噪声范围内。换句话说，dummy 模式现在可以作为 real-client 的"对等替代品"用了——代价是 dummy 端始终多付一份 H2D/D2H 拷贝，但换来了同主机多 vLLM 实例复用一份 RDMA QP/NIC 资源的能力，并让 KV 缓存在 vLLM 进程重启后继续存活。

2. 上游 commit 分析（huangyibo@421fedc）

该 commit 在 MooncakeStoreConfig 里加了 enable_dummy_client / real_client_address 两个字段，并新加了一个 _setup_mooncake_store 来分别走 store.setup_dummy(...) 和 store.setup(...)。配套加了 3 个测试。问题是这个 commit 原样合并是跑不起来的：

Bug 1 — AttributeError
_setup_mooncake_store 在 dummy 路径里打印日志时引用了 store_config.client_server_address，但 dataclass 字段是 real_client_address。任何 dummy 启动都会在调用 setup_dummy 之前崩。
Bug 2 — 测试与实现不一致

Bug 3 — 漏写 f-string
"Initialize MooncakeDistributedStore failed in {mode} mode."—— 没有 f 前缀，{mode} 会被原样打印。
修复 移植到 vllm/distributed/kv_transfer/kv_connector/v1/mooncake/mooncake_store_worker.py 时，统一用 enable_dummy_client / real_client_address 命名，并加 MOONCAKE_ENABLE_DUMMY_CLIENT / MOONCAKE_REAL_CLIENT_ADDRESS 环境变量覆盖路径，校验缺失地址时 ValueError。所有现存的 30 个 worker 单元测试通过。

3. 两种模式的数据流

Real-client 模式（当前默认）

每个 vLLM worker 进程自己持有 RDMA QP，KV cache 直接通过 GPUDirect / pinned memory 在 GPU ↔ Mooncake Store 间传输。GPU 指针可以直接调 store.register_buffer，因为 NIC 通过 ibverbs 注册得到对 GPU 内存的 DMA 访问权限。

Dummy-client 模式（新加入）

vLLM worker 通过 RPC/IPC 连到本机一个单独的 mooncake_client 进程（real client owner），后者集中持有 RDMA 资源。多个 vLLM 实例可以共用同一份 RDMA QP/NIC，且 KV 缓存能在 vLLM 重启后保留。代价是 vLLM 这一侧不能再直接把 GPU 指针交给 store。

4. SGLang 是怎么处理这个的

我们查了 SGLang PR #4880 和 Mooncake HiCache 集成文档。核心结论：SGLang 并没有什么 GPU-IPC 黑魔法，它只是不在 dummy 模式下注册 GPU 指针。

SGLang HiCache 是 3 层架构：

层位置分配器角色
L1GPU 显存vanilla cudaMalloc当前计算上下文
L2主机内存MooncakeHostTensorAllocator缓存淘汰的中间层
L3Mooncake 远端段master 管理跨节点持久化

关键点：standalone（dummy）模式要求 L2 分配器必须是 MooncakeHostTensorAllocator，它把 L2 的所有 host tensor 都直接从 dummy client 的 SHM 池里切出来。所以 register_buffer(host_tensor.data_ptr(), size) 调用时，buffer 本来就在 SHM 区域内，能通过 shm_helper_->get_shm() 查找。

L2 ↔ L3 是零拷贝（bytes 已经在 Mooncake 的 SHM 里），只有 L1 ↔ L2 需要 cudaMemcpy。

vLLM 没有 L2 这一层。vLLM 的 MooncakeStoreConnector 直接把 GPU KV cache evict 到 Mooncake，没有 host-tier 中间层。所以不能照搬 SGLang —— 要么补一个 L2（大改），要么在 connector 里做一个轻量级的 staging ring。

5. 我们的实现：SHM 中转环

选择借鉴 SGLang 的"通过 SHM 中转"的思路，但不引入完整的 L2 层，而是在 MooncakeStoreWorker 内部加一个有界的 staging ring：

核心组件

_DummyStagingPool
  • 启动时 alloc_from_mem_pool(N) 从 dummy client 切一块 SHM
  • 调一次 register_buffer(base, N) 注册到 owner
  • 调 cudaHostRegister 锁页让异步拷贝能真正流水
  • 切成 N / sum(block_len) 个 slot，acquire/release 受 Condition 保护
_StagingCopier
  • 线程级单例：每个 send/recv thread 一份
  • 自带一个 dedicated torch.cuda.Stream
  • issue() 单段 cudaMemcpyAsync
  • sync() 每 batch 同步一次该流
  • 没锁页时优雅降级为同步 cudaMemcpy

Put 路径（卸载）

slots = staging_pool.acquire(len(keys))
for k, (gpu_addrs, sizes) in enumerate(zip(addrs_in, sizes_in)):
    slot = staging_pool.slot_addr(slots[k]); off = 0
    for src, sz in zip(gpu_addrs, sizes):
        copier.issue(slot + off, src, sz)   # D2H 异步入队
        off += sz
copier.sync()                                # 单次流同步
res = store.batch_put_from_multi_buffers(keys, staged_addrs, sizes_in)
staging_pool.release(slots)                  # finally

Get 路径（取回）

slots = staging_pool.acquire(len(keys))
staged = [[slot_addr(slots[k]) + cumoff for cumoff in offsets(sz)] for k, sz in enumerate(sizes)]
res = store.batch_get_into_multi_buffers(keys, staged, sizes)   # 写入 SHM
for k, rc in enumerate(res):
    if rc < 0: continue                       # 失败的 key 跳过
    for src, dst, sz in zip(staged[k], gpu_addrs[k], sizes[k]):
        copier.issue(dst, src, sz)            # H2D 异步
copier.sync()                                  # 单次流同步
staging_pool.release(slots)

6. 性能优化：sync → async + pinned

第一版用同步 cudaMemcpy，结果吞吐 −42%。原因是每个 block 有 32 个分段（每层一段，每段 64 KiB），每个段一次 cudaMemcpy 调用，而 sync API 的固定开销 ~10 µs/次，于是 32 段 ≈ 370 µs/块——拷贝带宽根本没拉满，纯 API overhead。

我们在小基准里直接量了三种组合：

每块 stage 成本（µs，2 MiB / 32 段，越低越好）

初版（sync, pageable） async + pageable async + pinned ✓
数据由 .venv/bin/python 直接调 ctypes 测得；GPU = H100 / PCIe Gen5；2 MiB 共 32 个 64 KiB 段。

关键发现

  • Pageable host 强制每个 async 调用变同步（CUDA driver 自己用一个内置 pinned buffer 中转）。所以光改成 cudaMemcpyAsync 不够。
  • 必须把 SHM 池一开始就 cudaHostRegister 锁页，async 才能真正流水化。
  • 4 GiB 锁页内存对大内存机器没压力；如果失败，_StagingCopier 自动 fallback 到同步路径，不会崩。

8. 已知限制与后续

本次未覆盖

  • 外部命中率 = 0%：当前 workload 工作集 < GPU KV cache，所以 Mooncake 真正取回路径没被压到。"dummy 是否值得"的下一关测试应该用小 --gpu-memory-utilization 或更大 working set 来逼迫 evict 到 L2。
  • 没有跨节点测试。本次 owner 与 vLLM 都在同主机；跨节点 dummy → 远端 owner 还没量。
  • 暂时只验证了 Llama-3.1-8B / FlashInfer / TP=1。70B 类模型 block 会到 ~16 MiB，slot 数从 2048 缩到 ~256，可能要在更大 batch 下盯一下 staging-pool 反压。

如果要 ship 还需要

  • 跑一组真会触发 external hit 的 workload（小 GPU 缓存 + 大 corpus），把"dummy 模式实际 offload 收益 vs 它的拷贝开销" 量化出来。
  • 把 _DummyStagingPool / _StagingCopier 写测试（mock store 即可）。
  • 评估是否需要让 staging slot 大小可配置（独立于 local_buffer_size），尤其大模型场景。
  • 考虑给 P2PHANDSHAKE owner 的启动脚本（start_mooncake_owner.sh）补 --rpc-bind-host 之类的旋钮，避免再踩 --host=IP:PORT 让 RPC 起不来这种坑。

本次调试中的两个非显然的坑

  1. RPC TCP 监听起不来：原 start_mooncake_owner.sh 把 --host="$IP:$SEGMENT_PORT" 传给 mooncake_client，但同一个 --host 也被 coro_rpc_server 当 bind host 用，asio 解析 "192.168.0.116:50053" 作为 hostname 失败 → RPC 端口 50052 根本没起来。Dummy mode 之前没人用过 RPC 路径，所以这个 bug 一直潜伏着。已通过 --host=0.0.0.0 手动启动绕开
  2. Pageable host + cudaMemcpyAsync 实际还是同步：CUDA driver 文档明说，对 pageable host 内存，async 拷贝内部还是要先经过一份 pinned buffer，等价于同步。只有 cudaHostRegister 之后才能真异步。已通过 pool 初始化时锁页解决
