# 安全策略 (Security Policy)

## 报告安全问题

请务必通过 [漏洞提交表单](https://github.com/vllm-project/vllm/security/advisories/new) 私下报告安全问题。

## 问题分流 (Issue triage)

报告随后将由 [漏洞管理团队](https://docs.vllm.ai/en/latest/contributing/vulnerability_management.html) 进行分流处理。

## 威胁模型

关于 vLLM 的安全假设和建议，请参阅 [vLLM 文档中的安全指南](https://docs.vllm.ai/en/latest/usage/security.html) 以获取更多信息。

关于如何安全地与模型交互的更多信息和建议，请参阅 [PyTorch 的安全策略](https://github.com/pytorch/pytorch/blob/main/SECURITY.md)。

## 问题严重程度

我们将综合考虑过去处理问题的经验、受影响的版本、常见默认设置以及使用案例，来确定每个问题的风险。我们使用以下严重程度分类：

### 严重 (CRITICAL)

允许远程攻击者执行任意代码、完全控制系统，或在无需任何交互或权限的情况下显著破坏机密性、完整性或可用性的漏洞。示例包括通过网络的远程代码执行 (RCE)、允许利用链的反序列化问题。通常，这些问题的 CVSS 评分 ≥ 9.0。

### 高 (HIGH)

允许产生较大影响（如特定、受限环境下的 RCE 或重大数据丢失）但在利用时需要高级条件或一定信任度的严重安全缺陷。示例包括高级部署模式（如多节点）下的 RCE，或需要某种特权网络访问的高影响问题。这些问题的 CVSS 评分通常在 7.0 到 8.9 之间。

### 中 (MODERATE)

导致拒绝服务或部分中断，但不允许任意代码执行或数据泄露且影响有限的漏洞。这些问题的 CVSS 评分在 4.0 到 6.9 之间。

###  低 (LOW)

诸如信息泄露、日志错误、不可利用的缺陷，或需要本地/高权限访问且影响微乎其微的小问题。示例包括侧信道攻击或哈希碰撞。这些问题的 CVSS 评分通常小于 4.0。

## 提前通知策略

对于某些严重等级为 **严重**、**高** 或 **中** 的安全问题，我们可能会向发布 vLLM 的特定组织或供应商发出提前通知。此提前通知的目的是为了针对严重问题进行修复程序的协调发布。

- 此提前通知将以私人邮件的形式发送。它也可能包括在发布前几天将安全联系人添加到 GitHub 安全通告中。
- 如果您希望加入提前通知组，请发送邮件并抄送 [漏洞管理团队](https://docs.vllm.ai/en/latest/contributing/vulnerability_management.html) 的所有成员。我们将逐案分析每个供应商联系人的资格。
- 发布或使用 vLLM 的组织和供应商，如果满足以下至少一项资格，则有资格加入提前通知组：
	- 利用上游 vLLM 项目的大规模内部部署。
	- 建立了内部安全团队并拥有全面的合规措施。
	- 对上游 vLLM 项目有积极且持续的贡献。
- 如果组织在问题公开之前发布修复程序或有关问题的任何其他信息，我们可能会取消其接收未来提前通知的资格。小组成员资格也可能根据纳入人员政策的完善而发生变化。
